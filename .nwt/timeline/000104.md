# 000104 — primary create 的伪 drop 导致 ERROR_ACCESS_DENIED 死循环

## 主席反馈
event 000103 fix 之后, 装新 setup.exe 启动 Flowntier, 还是
sidecar 卡死无限循环。

## 调查

新 sidecar (`9585af6a...`) 跑起来后 `~/Desktop/Flowntier.log`
显示 worker 0 持续一秒一次:
```
[TRACE] rpc pipe create failed; backing off
error: 鎷掔粷璁块棶銆 (os error 5)  ← ERROR_ACCESS_DENIED
worker=0 primary=true
```

`os error 5 = ERROR_ACCESS_DENIED` —— 与之前
"pipe 不存在" (`os error 2`) 是不同的错。

event 000103 fix B 我写的 primary 入口:
```rust
let is_primary = i == 0;
loop {
    let mut opts = ServerOptions::new();
    if is_primary {
        // Best-effort: if a previous pipe survived,
        // drop it so we can claim first instance.
        let _ = ServerOptions::new()
            .first_pipe_instance(false)
            .create(&p);     // ← 这是关键 bug
        opts.first_pipe_instance(true);
    } else {
        opts.first_pipe_instance(false);
    }
    let server = match opts.create(&p) { ... }
}
```

**Bug**: `ServerOptions::create(&p)` 不是 drop, 是**创建并
attach** 一个 server pipe 实例到现有 path。如果路径上已经
有 stale pipe (前一个 sidecar 残留), 这个调用**作为 secondary
instance attach 成功**, 然后下一行 `first=true.create(&p)`
被 OS 拒绝 —— 因为已经有 instance 持有这个 path, 不能再
声明 primary。

死锁循环: worker 0 永远拿不到 primary → backing off 1s →
重试 → 同样的失败。

Win32 named pipe 没有"删除 API"。要清掉 stale pipe 只有
两个办法:
- (a) 持有进程退出 (释放所有 handle)
- (b) Tauri shell 在 spawn 前 taskkill 旧 sidecar (event
  000103 fix C 已经做)

## 修复

event 000104: 删掉 worker 0 的伪 drop 逻辑, 直接调
`first_pipe_instance(is_primary).create(&p)`:
```rust
let is_primary = i == 0;
loop {
    let mut opts = ServerOptions::new();
    opts.first_pipe_instance(is_primary);
    let server = match opts.create(&p) {
        Ok(s) => s,
        Err(e) => {
            tracing::warn!(error = %e, worker = i, primary = is_primary,
                "[TRACE] rpc pipe create failed; backing off 2s");
            tokio::time::sleep(Duration::from_secs(2)).await;
            continue;
        }
    };
    ...
}
```

- worker 0 `first=true`: 第一次创建或 stale 已被 taskkill 杀掉
  后重新创建
- workers 1..N `first=false`: attach 到 worker 0 的 pipe
- 失败时 backing off **2s** (不是 1s), 减少日志噪音
  并给 OS 更多时间清理

注释说明: 仍然依赖 (a) 上游 taskkill 杀干净, 或 (b) 前次
sidecar 自己退出。如果两条都不满足, 我们只会 backing off,
不会 panic, 不会破坏 desktop shell 的事件流。
