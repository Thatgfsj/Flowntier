# 000089 — 深度静态分析: 完整问题清单 (27 issues)

## 主席说"不止五个，继续检查"
NWT 000088 列了 5 个 issues。**远远不够**。继续深挖代码后找到 **27 个问题**。

按严重度分组：

## 🔴 高严重度 (workflow hang 直接原因)

### Issue #1: `nwt_log` 工具存在但未注册
- `crates/agent-core/src/tool/nwt.rs` 436 行，定义了完整的 nwt 工具
- `ToolRegistry::with_builtins()` **不注册** nwt 工具
- 6 个 role 的 system prompt 都注入 NWT_INSTRUCTION 说"你有 nwt_log"
- **结果**: 模型调 nwt_log → ToolError::Other → 50 iterations 用完 → 
  MaxIterationsReached → status "FAILED: MaxIterationsReached(50)"
- **不修这个 + Issue #2 = 永久 hang**

### Issue #2: orchestrator status 匹配字面量太严格
- `orchestrator.rs:419` `matches!(s, "DONE" | "FAILED" | ...)` 
- 实际 status 是 `"FAILED: MaxIterationsReached(50)"` ≠ `"FAILED"`
- **结果**: 永不退出 → 5 min × 8 phase = 40 min

### Issue #16: `first_pipe_instance(true)` 只能调用一次
- worker 0 永远用 `first_pipe_instance(true)`
- 第一次 create OK，建立 pipe path
- 第一次 client disconnect 后，loop back 又 create(true)
- Windows: 第二次 `first_pipe_instance(true)` 会失败（已有 first instance）
- 错误时 sleep 1s 永远 backoff
- **结果**: worker 0 死掉 (但其他 15 个 worker 仍能用)
- 注释"first worker creates the pipe"误导，**实际只创建一次**

### Issue #18: 配额调度器发送 `__quota_retry__` 是假实现
- `scheduler.rs:8-15` 注释说"chief agent loop recognises __quota_retry__"
- **但代码里没人检查 `__quota_retry__`**
- 实际发送 `task: "(Quota refresh probe — ...)"` (人类可读字符串, 真的让模型回答)
- 模型随意回答任何东西 → 当作"API 通了" → clear_quota_failure
- **结果**: 5h 边界会浪费 token，**且不能真正检测配额是否恢复**

### Issue #27: 配额 probe 实现与注释不符
- 见 Issue #18 的代码 vs 注释差异
- 应该改成真实的"无 token 探测"（直接 HEAD `/v1/models`）

## 🟡 中严重度 (功能性问题)

### Issue #10: dispatch phase context 重复
- `orchestrator.rs:557` 把整个 PlanDoc 再传给 dispatch chief
- 上一轮 plan phase 已经把 PlanDoc 给过 chief
- **结果**: 浪费 token，chief 困惑（"我刚做了 plan，又要 review"）

### Issue #12: worker 数量为 0 时 Phase 6 拿空 context
- `orchestrator.rs:585-593` `workers_summary.join("\n\n")` 空 Vec → 空字符串
- critic 收到空 review context
- **结果**: 不可预测（可能 PASS 也可能 FAIL），silent failure

### Issue #14: bash tool timeout 错误用 `?` 传播
- `bash.rs:153` 把 Elapsed 错误用 `?` 传播
- 上层 `loop_.rs:230-233` 把它当作 is_error: true
- 这部分其实是 OK 的，**收回这个 issue**。但 bash tool 没有 cancel 的情况 timeout 触发时，会让 cmd 子进程被 kill_on_drop 杀掉
- 实际：没问题

### Issue #17: events_listener write_all 错误不 break
- `server.rs:213-214` `let _ = server.write_all(...).await;`
- pipe 断了也继续写，浪费 CPU
- **结果**: 不会 hang，但 CPU spike

### Issue #20: events broadcast Lagged 永久丢失事件
- `server.rs:216` `Lagged(_) => continue` 直接跳到下一个 recv
- 慢订阅者（busy UI）丢的事件**永远拿不回来**
- **结果**: UI 看到 phase 跳变，text 缺失

### Issue #21: events_listener 写错误不 break 内层循环
- 同 Issue #17

### Issue #22: list_models live fetch 缓存 1 小时
- 对 mimo: 第一次 GET /v1/models 成功 → 缓存 1h
- 之后 1h 内即使 mimo 模型列表变了也用旧缓存
- **结果**: 模型列表过时

### Issue #23: `fallback_chain` 读但不传
- `handlers.rs:1888-1897` ResolvedRole 包含 fallback_chain
- `orchestrator.rs:363-373` run_agent **不用** fallback_chain
- chairman 配置了 fallback_chain 也**完全没用**
- **结果**: mimo 失败时没有 fallback → 整个 workflow 死

### Issue #25: pipe_request 读响应无超时
- `lib.rs:77-83` `conn.read_exact` byte-by-byte，没有 timeout
- 服务端 hang 的话 Tauri 命令永远不返回
- **结果**: UI 永久显示 "发送中…"

### Issue #26: pipe_request byte-by-byte 读低效
- `lib.rs:77-83` 1 byte / `read_exact` 系统调用
- 1 MiB 响应 = 1M 次 read
- **结果**: CPU 浪费 10x
