# 000103 — Sidecar 静默 crash 卡死 47s — 三重 root cause + 修复

## 主席反馈
"软件启动后卡死 过了很长时间现在好了" / "AI 的 API 全部没有了，
同时也保存不了保存失败：pipe open \\.\pipe\flowntier_runtime:
系统找不到指定的文件" / "0x000000000090EEB8 指令引用了
0x000000000090EEB8 内存。该内存不能为 written"

## 调查

抓 `C:/Users/thatg/AppData/Roaming/flowntier/logs/flowntier.log`
和 Application Event Log, 看到 3 个独立的 bug 协同导致 47s 卡死:

### Bug A: logs::init typo + 无 panic hook
- `crates/pipe-server/src/logs.rs:63` 默认 log 路径写的是
  `home.join("Desktop").join("Flwntier.log")` —— 少一个 "o"。
  不是导致 crash 的根因, 但 chairman 找不到 log 文件做诊断。
- sidecar 启动期如果发生 panic, stderr 和 stdout 都空,
  没有任何线索。Application Error event 报 0xC0000005 + 
  偏移 0x000000000090EEB8 (位于 .idata section 范围内, 
  即 import resolution 区域)。这暗示 loader 在解析某个 import
  时遇到未初始化的 IAT slot。

### Bug B: 全部 RPC worker 用 first_pipe_instance(false)
- `crates/pipe-server/src/server.rs:157` 全部 16 个 RPC worker
  都 `ServerOptions::new().first_pipe_instance(false)`。
  Windows named pipe 协议要求**至少一个进程持有 
  FIRST_PIPE_INSTANCE 标志**来创建 pipe 路径; 其它都是
  SECONDARY 实例 attach。全部 first=false → pipe 永远不创建,
 客户端连接永远 ERROR_FILE_NOT_FOUND (os error 2)。

- 旧注释 (event 000094) 说 "tokio ServerOptions::create 
  already creates the pipe on first call" —— **错的**。tokio
  只是透传 first flag 给 Win32 API, 不会自动设 true。

### Bug C: spawn_runtime_sidecar 静默吞 taskkill 失败 + 无超时警告
- `apps/desktop/src-tauri/src/lib.rs:182` taskkill /f /im 用错了
  进程名 (`flowntier-runtime.exe` 连字符), 真实 sidecar binary 是
  `flowntier_runtime.exe` 下划线。taskkill 返回 128 (no match),
  代码忽略 exit code, sleep 1s 后照常 spawn。
- 30s "wait for sidecar healthy" 循环超时后**默默返回**, 没
  eprintln, 没 tracing error, 没让 webview 知道 sidecar 没起来。
  这就是 chairman 看到 47s 卡死 (47s = 30s spawn + 17s 后续重试) 后
  "过了很长时间现在好了" 的原因。

## 修复

### Fix A: logs typo + panic hook
- `logs.rs:63` Flwntier → **Flowntier** (修正拼写)
- `logs::init` 不再用 `.expect()` — 桌面 read-only / OneDrive 同步
  冲突时 fallback 到 stderr-only
- `flowntier-runtime.rs` 顶部 `std::panic::set_hook`:
  - panic 时 eprintln 全部 info + `std::backtrace::Backtrace::force_capture()`
  - 同步写到 `~/Desktop/Flowntier.log` (best-effort)

### Fix B: pipe first_instance 修复
- `server.rs` RPC worker 0 用 `first_pipe_instance(true)` (primary)
- workers 1..N 用 `false` (secondary attach)
- 进入 primary 模式前, 先用 `first=false` 的 ServerOptions::create
  做 best-effort drop, 避免 stale pipe 阻塞

### Fix C: spawn 错误可见 + 正确进程名
- `lib.rs` taskkill 进程名改为 `flowntier_runtime.exe` (下划线)
- taskkill 退出码用 `tracing::warn` 输出 (128 = no match 也记录)
- 30s 超时后 `tracing::error!` 提示看 `~/Desktop/Flowntier.log`
- sidecar 的 stdout/stderr 通过 `CommandEvent::Stdout/Stderr` 转发
  到 `tracing::info!/error!`, 不再静默丢失
