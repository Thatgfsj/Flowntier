## Issue #1: `nwt_log` 工具不存在 (潜在 high)

**问题**：`ToolRegistry::with_builtins()` (`crates/agent-core/src/tool/mod.rs:203-211`)
只注册了 5 个工具：`bash, read, write, patch, grep`。

但所有 6 个 role 的 system prompt (`crates/agent-core/src/prompt/mod.rs:368-382`) 
都注入了这段 NWT_INSTRUCTION:
```
你有一个工具 `nwt_log` 可以把事件记到项目根目录的 `.nwt/timeline/` 里
```

**后果**：
- 模型看到 prompt 说有 nwt_log
- 模型尝试调用 nwt_log
- `tools.execute("nwt_log", ...)` → `ToolError::Other("unknown tool: nwt_log")`
- 每次 nwt_log 失败都消耗一次 `max_iterations: 50`
- 同一 args 连续 3 次失败 → `ABORTED_REPEAT` (good case)
- 不同 args → 永远不会 abort → 50 次用完 → `MaxIterationsReached` →
  status = `"FAILED: MaxIterationsReached(50)"` 
- 主席的 `run_agent` 用 `matches!()` 严格匹配字面量 `"FAILED"`，**匹配不上** → 永不退出
- 5 min × 8 phase = 40 min 死锁

**修复建议**（3 个选项，主席挑）：
- A. 修 model 行为：在 NWT_INSTRUCTION 改成 "不要调用 nwt_log，目前还没实现"
- B. 修 run_agent 的状态匹配：`last_status.starts_with("FAILED")` 或类似
- C. 实际实现 nwt_log 工具（最完整但工作量大）

我推荐 A+B 都改，B 是真正的状态机 bug。

## Issue #2: orchestrator 状态匹配太严格 (high)

`crates/pipe-server/src/orchestrator.rs:419`:
```rust
if matches!(last_status.as_str(), "DONE" | "FAILED" | "ABORTED" | "ABORTED_REPEAT" | "TIMEOUT (300s)") {
    return false;
}
```

`matches!` 在字符串上做字面量匹配。`"FAILED: DNS error"` **不等于** `"FAILED"`，
所以 `run_agent` 收到任何带前缀的失败 status 都不会退出 → 5 min 强制 timeout。

**修复**：改成
```rust
if last_status == "DONE" 
   || last_status.starts_with("FAILED") 
   || last_status.starts_with("ABORTED")
   || last_status.starts_with("TIMEOUT") {
    return false;
}
```

## Issue #4: chief 看到的工具列表和 prompt 矛盾 (low)

`crates/agent-core/src/prompt/mod.rs:142`:
```
- 你没有文件工具。如果需要看代码，让「实施」去做
```

但 `crates/agent-core/src/loop_.rs:158` 把 5 个工具 schema 全注入到 chief 的 system prompt:
```rust
Message::system(derive_system(&agent_id, tools.schemas())),
```

`derive_system` 把 `tools.schemas()` (含 read/write/patch/grep) 塞进 chief prompt。
chief 看到 prompt 说"没文件工具" + "你有 read/write/patch/grep" — 矛盾。

**修复建议**：
- A. 根据 role 过滤 schema（chief 只看到 bash, 没有 file 类）
- B. 修改 chief prompt 改成 "你原则上不用文件工具，但有 read 必要时可用"

## Issue #5: 工具调用的 capabilities 限制缺失 (low)

`crates/agent-core/src/loop_.rs:162-167`:
```rust
let tool_ctx = ToolContext {
    workspace: workspace.clone(),
    approved: true,                              // ← 永远 true
    capabilities: crate::tool::Capabilities::default(),  // ← 永远默认
    cancel: Some(cancel.clone()),
};
```

BugHunter/Reviewer (read-only critic) 应该用 `Capabilities::read_only()`，
但 orchestrator 一直传 `default()` = 全部能力。Critic 能写文件/执行命令。

**修复建议**：根据 spec.role 选不同的 capabilities。
