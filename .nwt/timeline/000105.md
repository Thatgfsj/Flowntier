# 000105 — 用 OS 默认行为替代 first_pipe_instance primary 设计

## 主席反馈
event 000104 修了 worker 0 伪 drop, 但启动后还是 worker 0
一直 ERROR_ACCESS_DENIED (os error 5), 其他 worker 都
"rpc pipe instance created"。桌面和 sidecar 都活, 但只有
worker 0 的 primary 失败, 永不恢复。

## 调查

计数 (~/Desktop/Flowntier.log 写入 sidecar stderr 转发的
tracing::error!, + 安装 log):
- primary=true **create failed**: 165 (worker 0 一直 fail)
- "rpc pipe instance created": 82 (其他 15 个 worker 都成功)
- "client connected": 59

矛盾: 同一进程内 worker 0 fail, worker 1..15 都 create 成功。

## Root cause

tokio 1.x 的 `ServerOptions::create()` 在默认 `max_instances
= unlimited` 下, **每次调用都创建一个独立的 primary
instance**, 不论 `first_pipe_instance` 设什么。`first=true`
只影响**当 pipe 已存在**时能否 attach。

race:
1. tokio::spawn 16 个 worker task, 不保证 spawn 顺序
2. worker 1..15 几乎同时调用 create(first=false) → OS 每个
   都创建一个独立 primary instance (max_instances=unlimited)
3. worker 0 终于轮到, 调 create(first=true) → OS 发现已经
   有 N 个 instance, 拒绝 (ERROR_ACCESS_DENIED)
4. 死循环 backing off 2s, 永不恢复

`max_instances` 默认是 PIPE_UNLIMITED_INSTANCES (255),
不是 OS 限制的 "primary + 多个 secondary" 的常见误解。

event 000094 的注释 "tokio ServerOptions::create already
creates the pipe on first call" 是错的。

## 修复

删掉 first_pipe_instance 标志, 所有 worker 用 OS 默认行为:
- max_instances = 255 (默认)
- 16 个 worker 每个独立创建 primary instance
- 客户端连接时, kernel 在所有 instance 间 round-robin
- **完全等价**于之前的"primary + secondary"模型的负载均衡
- 没 race, 没 ACCESS_DENIED

代码改动:
- `crates/pipe-server/src/server.rs` rpc_listener: 删掉
  `let is_primary = i == 0; opts.first_pipe_instance(is_primary)`,
  所有 worker 用 `ServerOptions::new().create(&p)` (无 flags)
- 顺手把 tracing 里的 `primary = is_primary` 字段也去掉
- events_listener 已经用 first=false, 保持不变

## 后续根因

如果 taskkill 之后 OS 仍然持有 stale pipe handle, 需要:
- 000106 候选: spawn 前用 `WaitNamedPipeA("\\\\.\\pipe\\xxx",
  NMPWAIT_USE_DEFAULT_WAIT)` + `CloseHandle` 强制释放
- 000107 候选: 给 pipe 设 unique 名 (用 PID), 完全避开
  全局命名空间冲突

但当前用 max_instances=unlimited 之后, **即使有 stale pipe,
16 个 worker 也会重新创建自己的 instance**, 不再被 race
卡住。所以 000106/000107 是优化, 不是必要修复。
