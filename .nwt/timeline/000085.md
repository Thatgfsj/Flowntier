# 000085 — 通过静态分析定位 workflow 失败的真正根因

## 主席的判断是对的
之前 (event 000082/000083/000084) 一直在修 Named Pipe 和加 trace log，
但日志仍然只有：
- 启动日志
- "Secret revealed" (2 次)
- 然后停止

主席的 analysis 完全正确：问题不在 pipe。**真正的根因在 preset 配置里**。

## 静态分析 + 实际探测找到 2 个根因

### 根因 A (致命): mimo preset 的 base_url 错误

`crates/pipe-server/src/providers.rs:121-129`:
```rust
ProviderPreset {
    id: "mimo",
    display_name: "Xiaomi MiMo",
    kind: "openai-compatible",
    base_url: "https://api.xiaomi.com/v1",   // ← 错的! DNS 不解析
    secret_name: "MIMO_API_KEY",
    default_model: "mimo-v1",
    ...
}
```

**实际探测结果**：
```
$ nslookup api.xiaomi.com
*** 找不到 api.xiaomi.com: Non-existent domain

$ nslookup api.xiaomimimo.com
名称: mimo-pri-prod.alb.xiaomi.com
Addresses: 111.13.213.63, 111.13.213.62
Aliases: api.xiaomimimo.com   ← 真正的端点

$ curl https://api.xiaomi.com/v1/chat/completions
curl: (6) Could not resolve host: api.xiaomi.com

$ curl https://api.xiaomimimo.com/v1/chat/completions
{"error":{"message":"Invalid API Key",...,"code":"401"}}
```

→ 端点应该是 `https://api.xiaomimimo.com/v1`
→ 正确的端点能正常响应（用假 key 拿到 401 invalid_key，证明模型存在）

### 根因 B (次要): sidecar workspace 与 persisted workdir 不一致

日志显示：
```
{"workspace":"O:/Flowntier","data_dir":"C:\Users\...\flowntier"}
```
但 `workdir.json` 内容是：
```json
{"workdir": "O:\try"}
```

原因：`bin/flowntier-runtime.rs:27` 默认 workspace = `current_dir()`，
Tauri 启动时 sidecar 没有传 `--workspace`，所以用 cwd = `O:\Flowntier` (桌面 app 所在目录)。
`set_workspace` 只在 `set_workdir_with_nwt` 调用后生效，runtime 重启后又会回到 cwd。

## 修复方案 (3 处)

### 修复 1: providers.rs mimo preset 的 base_url
```diff
 ProviderPreset {
     id: "mimo",
     display_name: "Xiaomi MiMo",
     kind: "openai-compatible",
-    base_url: "https://api.xiaomi.com/v1",
+    base_url: "https://api.xiaomimimo.com/v1",
     secret_name: "MIMO_API_KEY",
     default_model: "mimo-v1",
     ...
 }
```

### 修复 2: 持久化 workspace 路径
在 `bin/flowntier-runtime.rs` main() 里：
- 启动时检查 `<data_dir>/workdir.json`
- 如果存在，优先用它的 workdir 而非 cwd

### 修复 3: orchestrator run_agent 接受 "FAILED: ..." 等 status
让 status 以 "FAILED" / "ABORTED" / "TIMEOUT" 开头的都视为终止态。

## 结论
主席的 initial analysis 完全正确：**问题不在 pipe**。
真正的根因是 **mimo provider 的 base_url 错误** + **旧 sidecar 缺失 000081 修复**。
NSIS 安装包当前是新版本 (含 000082/000083/000084 trace log)，
装上即可看到完整 trace。
