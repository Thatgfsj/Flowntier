# 000087 — 修复交付失败：NSIS 安装的是 MSVC 旧二进制

## 主席说"还有问题，我无法使用"
即使打了 10:45 的安装包，运行时仍然用 `https://api.xiaomi.com/v1`（错的）。
说明安装的二进制不是新版的。

## 静态分析 + 文件系统验证找到根因

### 根因：NSIS installer 引用错路径

`target/release/nsis/x64/installer.nsi:689`:
```nsi
File /a "/oname=flowntier_runtime.exe" \
  "O:\clawwork\Flowntier\apps\desktop\src-tauri\binaries\flowntier_runtime-x86_64-pc-windows-msvc.exe"
```

→ NSIS 编译时把 **`-msvc.exe`** 文件打包进安装包。
→ 这台机器没装 MSVC 工具链（`cl.exe` missing），所以 msvc.exe 从来没重新构建过。
→ `apps/desktop/src-tauri/binaries/flowntier_runtime-x86_64-pc-windows-msvc.exe`
  是 **Jul 1 23:27** 的老文件，MD5 `33f2550abe1f...` (3 周前的版本)
→ 我用 `--target x86_64-pc-windows-gnu` 重新构建的二进制只更新了 `-gnu.exe` 路径
→ `tauri build` 不带 `--target` → 默认 MSVC → 旧 msvc.exe 被打包 → 安装包是旧的

### 已安装的运行时（旧版）— 用户实际跑的

| 文件 | 时间 | MD5 |
|------|------|-----|
| `O:\Flowntier\flowntier_runtime.exe` (实际安装) | Jul 15 01:04 | `f69ccfd4...` |
| `binaries\flowntier_runtime-x86_64-pc-windows-msvc.exe` (源) | Jul 1 23:27 | `33f2550a...` |
| `binaries\flowntier_runtime-x86_64-pc-windows-gnu.exe` (新) | Jul 15 10:39 | `6394df9d...` |

`f69ccfd4` 这个 MD5 在 `target/` 任何地方都找不到——说明这是 NSIS 重新打包时
从旧 msvc.exe 提取出来放到 install 目录的中间产物。 

## 修复

**根因修复**：把新 GNU 字节复制到 MSVC 路径，让 NSIS 打包正确的二进制：
```bash
cp target/x86_64-pc-windows-gnu/release/flowntier-runtime.exe \
   apps/desktop/src-tauri/binaries/flowntier_runtime-x86_64-pc-windows-msvc.exe
```

**验证**：MSVC binary 现在有 `https://api.xiaomimimo.com/v1`（正确的端点），
没有 `api.xiaomi.com`。
两个路径的 MD5 现在都是 `6394df9da33c1f6aa1f94538d17af115`。

## 长期防御（建议）

加一个 pre-build 检查脚本（在 `pnpm tauri build` 之前强制执行），
如果 `-msvc.exe` 的内容比 `-gnu.exe` 旧就报警。
否则 `cargo build --target x86_64-pc-windows-gnu` 不会自动同步 MSVC 路径。
