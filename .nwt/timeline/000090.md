## Issue #28-#40 (续)

### #28 — `log_webview_console` 不区分级别 [low]
`lib.rs:369-371`: 收到 webview console 后 `tracing::info!`
不管 `console.error` / `console.warn` / `console.log` 都打 INFO。
- 严重度信息丢失
- 修复: 解析 line 前缀 (`err ` / `warn ` / `log `) 选不同 tracing level

### #29 — `quota_failures.attempt_count` 永远递增 [medium]
`storage/lib.rs:815-829` `record_quota_failure` ON CONFLICT DO UPDATE
只把 `status` 设成 `'failed'`，**永不变成 `'pending_5h_wait'`**。
- scheduler 找 `pending_5h_wait` 永远查不到
- 5h 刷新机制 dead code

### #30 — 配额 probe 失败也写 quota_failures [medium]
`handlers.rs:2125-2129` `record_quota_failure` 在 `status_clean != "DONE"`
时调用。但 5h 边界的 probe 失败也会被记成 quota 失败。
- 修复: 在 record_quota_failure 调用前检查 `task` 是否为 `__quota_retry__`
- 或者: 真正的 probe 不通过 `run_task` 走，绕过 quota tracker

### #31 — `OpenAiProvider::compat()` 不验证 URL [low]
`orchestrator.rs:368-372` 用 `compat()` 而非 `compat_checked()`。
坏 URL 要等 HTTP 发送时才报错。
- 修复: 改用 `compat_checked()`

### #32 — `cancel_workflow` 是 stub [medium]
`handlers.rs:1257-1263` `POST /api/workflow/{id}/cancel` 永远返回
`"no active workflow to cancel"` 无论有没有 workflow。
- **主席无法取消正在跑的 workflow**
- 修复: 维护 active workflow map + cancel token

### #33 — `cancel_workflow` Tauri 命令没 URL 编码 wf_id [low]
`lib.rs:666-673` `format!("/api/workflow/{}/cancel", id)` 直接拼。
- `wf_id` 含 `/` 或特殊字符会破坏路由
- 修复: 用 `url_encode(id)`

### #34 — HTTP bridge `127.0.0.1:8765` 无认证 [HIGH, 安全]
`bin/flowntier-runtime.rs:136-143` 启动 HTTP bridge。
`ws_bridge.rs` CORS `*` + 无 token。
- **任何**本机进程（包括恶意 browser tab、download、user-installed malware）
  都能 POST `/rpc` 调用：
  - `GET /api/settings/secrets/{name}/reveal` → 读所有 API key 明文
  - 任意 `run_task` / `run_workflow`
  - 改 `role_overrides`, workspace, providers
- 修复: 加 token auth (从 environment 读 secret) 或 OS keychain

### #35 — `OpenAiProvider::compat` 接收 `api_key.to_string()` 丢 Zeroizing [low]
`orchestrator.rs:366, 371` `resolved.api_key.to_string()` 拷出 plain String。
Zeroizing<String> 的目的（minimize lifetime）被破坏。
- API key 在堆上活到 provider 被 drop
- 修复: 让 OpenAiProvider 接受 Zeroizing<String> 或内部用 zeroize::Zeroizing

### #36 — broadcast::channel(1024) 太小 [medium]
`handlers.rs:90` broadcast buffer 1024。
一个 agent run 50 iter × 200 tokens × 多个 TextDelta events = 10K events。
- 慢 UI → events 永久丢失 (Lagged)
- 修复: 改 8192 或 16384

### #37 — 取消 workflow 也算 quota failure [low]
`handlers.rs:2112` `status_clean != "DONE"` 包括 `ABORTED`。
- 用户关 window → agent 收到 cancel → ABORTED → 记 quota failure
- 修复: 排除 `ABORTED` / `ABORTED_REPEAT` status

### #38 — `tools.schemas()` 调用 2 次 [low]
`loop_.rs:154, 158` cached + re-compute。
- 启动多 1 次 schemas() 序列化
- 修复: 用 cached 版本

### #39 — `derive_system` 3 次序列化 roundtrip [low]
schema → JSON → deserialize → embed template → re-serialize Message。
- 修复: 减少冗余序列化

### #40 — `pipe_request` 同步阻塞在 async context [medium]
`lib.rs:49-50` `ClientOptions::new().open(RPC_PIPE)` 是 **同步**。
在 async fn 里直接调，**阻塞 tokio worker thread**。
- 慢 pipe (stuck runtime) → 整个 Tauri 异步 runtime 卡 1 个 worker
- 修复: `tokio::task::spawn_blocking` 包装
